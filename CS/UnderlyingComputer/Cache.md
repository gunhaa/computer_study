# Chapter 5. 작은 것으로 큰 성과 이루기, 캐시

- 문제: CPU의 속도와 메모리의 속도가 일치하지 않는다
  - 문제가 된 속도를 해결하기 위해 캐시를 사용한다(register, L1, L2, L3)
    - 멀티코어 환경에서 캐시를 사용하면 코어 간 일관성 문제가 발생한다(CPU는 일관성을 유지하기 위해 MESI protocol을 사용하여 성능을 희생한다)
    - 소프트웨어적으로 데이터 가시성과 일관성을 완벽히 맞추기 위해서는 CPU 성능 최적화를 제한해야 하므로 속도를 희생해야 한다(명령어 재정렬 및 캐시 지연 쓰기를 방지하기 위해 고급 언어에서 volatile 키워드나 Memory Barrier 사용)
- (반드시)캐시 시간/공간적 지역성을 최적화 하기 전 반드시 분석도구를 사용하여 캐시의 적중률이 시스템의 병목이 될때만 최적화를 해야한다

## Section 5.3. 다중 스레드 성능 방해자

- 캐시는 `캐시 라인`이라는 이름으로 저장된다
  - `캐시 라인`이 캐시의 시간/공간적 지역성이 생기는 이유이다
  - `캐시 라인`의 묶음은 일반적으로 64byte이며, 캐시가 적중하지 못하면 이 묶음 데이터가 캐시에 저장된다

### 5.3.2 두 번째 성능 방해자: 캐시 튕김 문제

- 첫 번째 예제 프로그램
```cpp
atomic<int> a;

void threadf()
{
    for (int i = 0; i< 500000000; i++)
    {
        ++a;
    }
}

void run()
{
    thread t1 = thread(threadf);
    thread t2 = thread(threadf);

    t1.join();
    t2.join();
}
```

- 두 번째 예제 프로그램
```cpp
atomic<int> a;

void run()
{
    for (int i = 0; i < 1000000000; i++)
    {
        ++a;
    }
}
```

- 첫 번째 프로그램은 16초, 두 번째 프로그램은 8초가 걸린다
- linux의 perf를 이용해 분석해보면, 첫 번째 프로그램은 `0.15 insn per cycle`, 두 번째 프로그램은 `0.60 insn per cycle`이다
  - 첫 번째 프로그램은 하나의 클럭 주기 동안 기계 명령어가 0.15개 실행됬고, 두 번째 프로그램은 하나의 클럭 주기 동안 0.60개의 기계 명령어가 실행되었다
  - 다중 코어 컴퓨터를 사용했기에, 서로의 코어에 a를 사용하고 이것이 캐시되고 있어 atomic한 연산을 위해 a변수를 반드시 invalidation해야한다
    - 이로 인해 캐시 튕김이 발생하며, 다중 스레드 프로그램에서 더 많은 시간이 걸리게 된다
    - 이를 캐시 튕김(cache line bouncing) 혹은 캐시 핑퐁(cache ping-pong)이라고 한다
